# Writing Rules

Requirements must be Specific, Unambiguous, Testable, Feasible, Necessary, and Traceable.

## Language Standard
- **Prefer:** "The system shall..."
- **Example:** "The system shall allow an authorized Front Office Staff user to create a reservation."

## Avoid Vague Terms
Do NOT use subjective or vague terms unless defined with measurable criteria:
- Fast (Instead: "within 2 seconds under normal conditions")
- Easy
- User-friendly
- Secure
- Efficient
- Appropriate
- Sufficient

## Functional Requirement Components
Every major functional requirement should contain:
- Requirement ID
- Requirement Name
- Description
- Business Objective
- Actor
- Preconditions
- Trigger
- Input
- Main Flow
- Business Rules
- Output
- Alternative Flow
- Exception Flow
- Postconditions
- Acceptance Criteria

## Non-Functional Requirement Categories
NFRs should be measurable. Consider:
Performance, Scalability, Availability, Reliability, Security, Privacy, Maintainability, Usability, Accessibility, Compatibility, Observability, Disaster Recovery, Data Retention.
