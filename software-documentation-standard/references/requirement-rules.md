# Requirement Rules and ID Standard

Use stable, unique IDs across all documentation. Never reuse an ID.

## BRD IDs
- `BR-001`, `BR-002`

## PRD IDs
- `FEAT-001`, `FEAT-002`
- `US-001`, `US-002` (User Stories)
- `PRD-GOAL-001`

## SRS-FRS IDs
- `FR-001`, `FR-002` (Functional Requirements)
- `NFR-001`, `NFR-002` (Non-Functional Requirements)
- `UC-001`, `UC-002` (Use Cases)
- `INT-001`, `INT-002` (Integrations)
- `DATA-001` (Data Requirements)
- `NOTIF-001` (Notifications)
- `RPT-001` (Reporting)

## FSD IDs
- `FSD-001`, `FSD-002`
- `SCR-001`, `SCR-002` (Screens)
- `WF-001`, `WF-002` (Workflows)
- `API-001`, `API-002` (API Endpoints)

## Technical Design IDs
- `TD-001`, `TD-002`
- `ADR-001`, `ADR-002` (Architecture Decision Records)

## Testing IDs
- `TC-001`, `TC-002` (Test Cases)

## General Requirement Rules
1. One requirement per ID.
2. If a requirement is split, deprecate the old ID and create new ones.
3. If a requirement is deleted, do not reuse its ID.
