# SRS-FRS Standard

## Purpose
Define *what* the software must do to fulfill the product requirements. This is the bridge between the product vision and the detailed functional specifications.

## Must Contain:
- Functional Requirements (FRs)
- Non-Functional Requirements (NFRs)
- Business rules
- User roles & Permissions
- High-level use cases and workflows
- Data requirements (entities, not schemas)
- Integration requirements (what systems, not the exact API payloads)
- Notification & Reporting requirements
- High-level error handling requirements
- Acceptance criteria

## Must NOT Contain:
- Screen-by-screen field validations (save for FSD)
- Technical architecture (save for Technical Design)
